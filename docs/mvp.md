# MVP 设计

## 1. MVP 边界

### 1.1 用户与目标

唯一用户是你自己，因此系统目标不是“让所有策略都能接入”，而是：

- 可靠地承载你定义好的策略协议
- 用受控方式执行策略
- 让回测结果可解释、可复现、可逐步扩展

### 1.2 第一阶段要解决的问题

MVP 必须闭环以下链路：

1. 输入策略描述和规则要求
2. 生成符合协议的策略脚本
3. 校验策略脚本是否安全、是否符合接口
4. 将脚本加入本地策略库
5. 用指定参数执行日线回测
6. 输出净值、绩效指标、交易记录

### 1.3 第一阶段范围内

- 市场：A 股现货
- 资产：股票
- 方向：只做多
- 数据频率：日线 OHLCV + 基础状态字段
- 调仓频率：日级调仓，典型为每 2 到 3 个交易日
- 股票池：手工指定或由数据层提供静态/动态股票池
- 成交：支持停牌、涨跌停不可成交、交易成本、滑点
- 复权：默认以后复权价格做信号研究，成交价格使用原始交易价或可配置价
- 输出：收益率、年化、最大回撤、夏普、胜率、盈亏比、交易明细、净值序列

### 1.4 第一阶段范围外

- 高频和日内回测
- 多资产统一账户
- 保证金、杠杆、融券卖空
- 组合优化器
- 参数搜索平台
- Web UI 和任务中心
- 用户管理、权限和 API 服务

## 2. 系统架构

建议采用单机、本地文件驱动、模块边界清晰的结构。

### 2.1 模块划分

1. `protocol`
   - 定义 `BaseStrategy`、上下文对象、信号输出格式
   - 这是策略作者唯一应该依赖的接口层

2. `sandbox`
   - 负责脚本 AST 校验、白名单模块检查、危险调用拦截
   - 后续可扩展子进程隔离执行

3. `registry`
   - 负责策略入库、元数据记录、版本和校验摘要

4. `data`
   - 负责统一读取行情、交易状态、复权、成分股、基准等
   - 对外暴露回测所需的受控数据接口

5. `engine`
   - 负责调仓日调度、信号调用、订单生成、成交模拟、持仓更新、绩效统计

6. `reporting`
   - 负责指标计算、交易明细整理、净值输出

7. `cli`
   - 本地命令入口
   - 用于校验策略、注册策略、运行回测、查看模板

### 2.2 设计原则

- 策略脚本不能直接读任意文件、网络或系统资源
- 策略只能看到引擎传入的有限上下文
- 回测引擎掌控时间推进和成交逻辑
- 数据字段、执行时点、价格口径必须明确
- 所有回测运行都应该可以复现

### 2.3 推荐项目结构

```text
Trade/
├── docs/
├── examples/
├── strategies/
├── tests/
├── pyproject.toml
└── src/ashare_backtest/
    ├── cli/
    ├── data/
    ├── engine/
    ├── protocol/
    ├── registry/
    ├── reporting/
    └── sandbox/
```

建议把“数据文件”和“回测结果”继续放在仓库外，或后续放入 `storage/`、`results/` 这类目录，避免源码目录膨胀。

## 3. 受限 Python 策略协议

### 3.1 责任边界

策略脚本只负责：

- 声明策略元信息
- 在给定上下文中判断是否调仓
- 生成候选股票列表
- 给出目标权重建议

策略脚本不负责：

- 自行读取行情文件
- 自行撮合成交
- 自行维护账户现金和持仓
- 自行修改回测时间推进
- 进行网络请求或任意系统调用

### 3.2 最小接口

- `name`: 策略名称
- `description`: 简介
- `lookback_window`: 所需历史窗口长度
- `rebalance`: 是否在当前日期触发调仓
- `select`: 返回候选股票列表
- `allocate`: 返回目标权重映射

### 3.3 策略模板约束

- 仅允许导入 `math`、`statistics`、`typing`
- 禁止 `eval`、`exec`、`open`、`compile`、`__import__`
- 禁止访问双下划线属性
- 禁止 `while`、`try`、`with`、`lambda`
- 禁止用户脚本定义除策略类之外的大量自由函数和复杂副作用

MVP 先使用 AST 校验 + 受限执行环境。后续若需要更强隔离，再切到独立子进程甚至容器化执行。

## 4. 策略入库与回测流程

### 4.1 入库流程

1. 接收 AI 生成或手写的策略脚本
2. 执行 AST 校验
3. 验证是否存在唯一策略类，且继承 `BaseStrategy`
4. 验证必需属性和方法是否齐全
5. 计算脚本 hash
6. 将脚本复制到 `strategies/`
7. 写入本地元数据文件

### 4.2 回测执行流程

1. 读取回测配置
2. 读取行情和交易状态数据
3. 构建交易日历
4. 加载策略脚本并实例化
5. 逐个交易日推进
6. 若到达调仓日，则传入上下文给策略
7. 将目标权重转换为订单
8. 结合停牌、涨跌停、成本与滑点模拟成交
9. 更新持仓、现金和组合净值
10. 生成指标、交易记录和净值曲线数据

## 5. 常见量化陷阱处理

MVP 先明确机制，不追求一次性完全覆盖。

### 5.1 未来函数

- 策略只能访问当前调仓日之前可见的数据
- 引擎向策略暴露的数据窗口必须截断到当前时点
- 对财务数据和成分股数据，后续要保留“公告日/生效日”字段

### 5.2 幸存者偏差

- 股票池不能默认使用“当前仍上市股票全集”
- 动态股票池要支持按历史时点还原

### 5.3 复权口径

- 信号研究价格和成交价格必须分离
- 默认建议：
  - 信号使用后复权
  - 成交使用真实交易价口径

### 5.4 停牌与涨跌停

- 当日不可交易标的不能成交
- 回测结果中要保留未成交原因

## 6. 开发顺序建议

1. 先完成策略协议、校验器、注册流程
2. 再完成最小日线回测引擎
3. 再接一套稳定的数据适配格式
4. 最后再加策略生成、批量回测、结果可视化
