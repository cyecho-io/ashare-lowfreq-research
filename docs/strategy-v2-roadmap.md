# 策略 V2 路线图

这份路线图的目标不是罗列所有可能的想法，而是基于当前 `v1` 的真实问题，给出一个清晰、可执行的后续推进顺序。

当前 `v1` 已经解决了“有没有形成第一版策略”的问题。接下来 `v2` 要解决的是另一类问题：

- 这版策略到底稳不稳
- 这版收益到底靠什么在赚钱
- 这版约束是不是足够接近真实交易
- 这套流程能不能支持持续迭代，而不是每次重新试错

## 当前 V1 的主要问题

基于目前的回测和验证结果，`v1` 还存在以下四类核心问题。

### 1. 信号稳定性偏弱

- walk-forward 月度 IC 均值约为 `0.0067`
- 组合月收益与月度 IC 相关性不强

这意味着模型本身的排序能力还不够稳定，当前收益不完全来自一个强而稳定的预测信号。

### 2. 收益结构不够平滑

- 收益集中在少数月份
- 在部分月份出现明显回撤

这说明策略对市场阶段较敏感，还没有证明自己能在更广泛环境下稳定工作。

### 3. 风险约束仍然偏初级

当前只接入了：

- `max_names_per_industry=2`

但还没有系统处理：

- 单票容量
- 风格暴露
- 持仓集中度
- 成交冲击
- 组合容量上限

### 4. 执行层仍然偏理想化

虽然已经支持基础费用和滑点，但当前成交和容量约束仍然不足以代表真实执行。

尤其是流动性门槛一旦提升，收益会明显受损，这说明当前收益对小票或低流动性标的仍有依赖。

## V2 的核心目标

`v2` 不是简单地继续调参数，而是要完成三件更重要的事：

1. 证明策略在更长区间和更多市场状态下仍然有效
2. 搞清楚收益来源和风险来源
3. 把执行约束往真实交易推进一层

如果这三件事做完之后，策略依然成立，再谈新的特征、模型升级和版本扩张才更有意义。

## 短期计划

短期目标是先补“可信度”，而不是追求更高的回测收益。

### 1. 扩大 walk-forward 和回测验证区间

要做的事：

- 尽量把训练和回测区间往前扩展
- 在更多市场状态下验证 `v1`
- 对同一套配置做更长时间的复现

为什么先做这个：

- 目前样本区间偏短
- 当前收益有明显阶段性
- 不先扩区间，任何进一步优化都有过拟合风险

完成标准：

- 能得到一版更长周期的 `v1` 回测结果
- 能回答“这版策略是否只在当前样本期有效”

### 2. 补月度与年度分解报表

要做的事：

- 输出月度收益
- 输出年度收益
- 输出最大回撤区间
- 输出净值阶段拆解

为什么做这个：

- 当前只看总收益、Sharpe、回撤还不够
- 需要更细地看到收益集中在哪些阶段、风险出在哪些阶段

完成标准：

- 每次正式回测都能生成一份阶段分解报告

### 3. 固化版本归档结构

要做的事：

- 为 `v1`、后续实验和候选版本建立统一命名规范
- 约定配置、指标、结果目录和文档归档方式

为什么做这个：

- 现在还能靠记忆维护上下文
- 版本一多就会开始混乱

完成标准：

- 任意一个实验都能快速定位到：
  - 配置文件
  - 分数文件
  - 回测结果
  - 文档说明

## 中期计划

中期目标是补“解释力”和“风险认知”。

### 1. 做收益来源拆解

要做的事：

- 看组合收益来自哪些月份
- 看是否集中在某些行业、某些风格或某些小票暴露
- 统计行业权重和持仓分布

为什么做这个：

- 现在知道结果是好的
- 但还不知道收益到底是模型预测带来的，还是某类暴露偶然贡献的

完成标准：

- 能回答“这版策略主要靠什么赚钱”

### 2. 做风格与行业暴露分析

要做的事：

- 统计组合的行业分布
- 观察是否偏向高波动、小市值、高换手或某类主题股
- 对比回撤月份的暴露特征

为什么做这个：

- 行业约束只解决了最粗的一层集中度问题
- 真正的风险往往来自更隐性的风格暴露

完成标准：

- 能回答“这版策略主要承担了哪些显性和隐性风险”

### 3. 评估 `min_daily_amount` 的更合理接入方式

要做的事：

- 不再用简单静态阈值硬切
- 尝试按分位数、动态门槛或容量约束方式接入

为什么做这个：

- 当前简单流动性过滤会明显伤收益
- 这不代表流动性不重要，而是说明接法太粗

完成标准：

- 找到一种比“硬阈值过滤”更合理的流动性处理方式

## 长期计划

长期目标是把策略从研究原型逐步往“可交易系统”推进。

### 1. 补更真实的容量与冲击模型

要做的事：

- 对买卖量占当日成交额比例做约束
- 引入更真实的冲击成本近似
- 在组合层考虑容量上限

为什么做这个：

- 当前回测默认交易规模对市场影响有限
- 真实交易里，容量约束会直接决定策略是否可实施

完成标准：

- 能回答“这版策略在什么资金规模下仍然可做”

### 2. 提升模型层稳定性

要做的事：

- 重新审视标签设计
- 评估特征稳定性
- 观察是否要引入多期集成、模型集成或更稳健的训练方案

为什么做这个：

- 现在组合能赚钱，但模型本身的信号稳定性还不够强
- 如果底层信号不稳，策略上限会受限

完成标准：

- 月度 IC 稳定性相比 `v1` 有明确改善

### 3. 建立持续研究工作流

要做的事：

- 固化实验流程
- 固化版本命名
- 固化结果对比模板
- 形成“提出假设 -> 运行实验 -> 记录结论 -> 决定去留”的闭环

为什么做这个：

- 真正限制迭代速度的，往往不是模型能力，而是流程混乱

完成标准：

- 每次新版本迭代都能快速比较、快速淘汰、快速收口

## 建议的推进顺序

为了避免同时做太多事情，建议按下面顺序推进：

1. 先扩验证区间
2. 再做收益与风险拆解
3. 再做更真实的执行约束
4. 最后才做模型升级和 `v2` 参数重构

这个顺序的核心逻辑是：

- 先验证这版策略到底是否成立
- 再理解它为什么成立
- 再判断它在真实世界里是否还能成立

## 第一阶段建议的具体交付物

如果现在立刻开始做 `v2`，建议先以这三项作为第一阶段交付物：

### 交付物 1：更长区间回测结果

输出：

- 新的正式配置文件
- 更长区间的 walk-forward 指标
- 更长区间的回测结果目录
- 一份 IC 诊断记录：
  - [strategy-v2-ic-diagnosis.md](/Users/yongqiuwu/works/github/Trade/docs/strategy-v2-ic-diagnosis.md)

### 交付物 2：收益拆解报告

输出：

- 月度收益表
- 回撤区间表
- 行业分布或持仓分布摘要

### 交付物 3：风险约束评估记录

输出：

- 行业约束影响说明
- 流动性约束影响说明
- 下一轮准备采用的执行约束方向

## 一句话总结

`v1` 之后最重要的事，不是继续把回测数字往上抬，而是把这版策略从“看起来有效”推进到“知道它为什么有效，也知道它在哪些条件下会失效”。
